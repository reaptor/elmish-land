# Elmish Land 2.0

Version 2.0 brings Elmish Land onto **[Fable 5](https://www.nuget.org/packages/Fable/5.0.0)** ([announcement](https://fable.io/blog/2026/2026-02-27-Fable_5_release_candidate.html)) and the matching ecosystem majors — `Feliz 3`, `Fable.Elmish.HMR 9`, and `vite 8`. New projects created with `init` get these versions out of the box, and a new `upgrade` command moves existing projects across in one shot.

This is a **major version bump** because the underlying libraries contain breaking changes. Most Elmish Land applications will compile after running `upgrade`, but some will need adjustments — see [Breaking changes you may hit](#breaking-changes-you-may-hit) below before upgrading a real project.

## Fable 5 by default

The Fable dotnet tool installed by `dotnet elmish-land init` is now version 5. This is a major step forward for the F# → JavaScript compiler that powers every Elmish Land app. Existing 1.x projects were on Fable 4; 2.0 makes Fable 5 the default for new projects and gives existing projects a one-command path to move over.

## Feliz 3, Fable.Elmish.HMR 9, vite 8

Fable 5 lands together with several aligned ecosystem upgrades. New scaffolded projects pin the following majors:

| Package                | 1.1 | 2.0 |
| ---------------------- | --- | --- |
| Fable (dotnet tool)    | 4   | **5**   |
| `Feliz`                | 2   | **3**   |
| `Fable.Elmish.HMR`     | 8   | **9**   |
| `vite`                 | 7   | **8**   |

These were upgraded together because they're all part of the same release wave — Feliz 3 targets Fable 5, and Fable.Elmish.HMR 9 follows along.

### URL change handling moved to a subscription

As part of the upgrade, the scaffold's URL-change handling has been reworked. Where 1.1 wrapped the entire view tree in a `React.router` element, 2.0 uses an Elmish subscription:

```fsharp
let private urlChangeSubscription _model =
    let routeMode = Feliz.Router.RouteMode.Hash // or .Path, based on elmish-land.json
    [ [ "ElmishLand"; "UrlChanges" ],
      fun dispatch ->
          Feliz.Router.Router.subscribeToUrlChanges
              routeMode
              (Route.parse >> RouteChanged >> dispatch) ]

let subscribe model =
    Sub.batch [
        Sub.map "Shared" SharedMsg (Shared.subscriptions model.Shared)
        urlChangeSubscription model
        // ...
    ]
```

Two practical consequences:

- Your view is no longer nested inside an extra router element, which keeps the rendered React tree closer to what you wrote.
- A small vendored copy of Feliz Router lives at `.elmish-land/Base/Router.fs`. It's regenerated by `init`, `upgrade`, and `restore`, so you should not edit it by hand. The `Feliz.Router` NuGet dependency has been removed from the scaffold.

If you wrote any code that assumed the view was rendered inside a `React.router`, you'll want to revisit it after upgrading.

## Breaking changes you may hit

This section summarises the breaking changes coming in from each upgraded library. The list is not exhaustive — refer to each project's release notes for the full picture.

### Feliz 3

This is the most likely source of code edits when upgrading. The full migration guide is at [Feliz: Upgrading from v2 to v3](https://fable-hub.github.io/Feliz/api-docs/Upgrade); the items below cover everything on that page.

**Version requirements** — `dotnet elmish-land upgrade` handles these for you, but worth knowing:

- React must be **v19 or higher** (Elmish Land has shipped React 19 since 1.1, so you're already there).
- Fable must be updated to 5.0.
- The .NET SDK must be **10 or higher**; `upgrade` writes the latest installed SDK to `global.json`.

**Architectural shift.** Quoting the upgrade guide: *"A lot of F# wrapper magic was removed. React bindings now behave as close as possible to actual React functionality."* In practice, several Feliz APIs that used to do extra work behind the scenes now require an explicit second call.

**API changes:**

- **`React.memo` requires a paired `React.memoRender` call** to render the memoised component.

  ```fsharp
  // Before
  let MemoFunction = React.memo<{| text: string |}> (fun props -> ...)

  // After
  React.memoRender(MemoFunction, {| text = text |})
  ```

- **`React.lazy'` requires a paired `React.lazyRender` call.**

  ```fsharp
  // Before
  let LazyHello: LazyComponent<unit> = React.lazy'(fun () -> ...)

  // After
  React.lazyRender(LazyHello, ())
  ```

- **`React.context` has been reworked** to align more closely with React itself. The new shape uses `React.createContext`, `React.useContext`, and `Provider` directly.

**Namespace migrations.** Several F# interop utilities have moved into the `FsReact` namespace. Update any references to:

- `FsReact.createDisposable`
- `FsReact.useDisposable`
- `FsReact.useCancellationToken`

**Component naming convention.** The built-in component values are now PascalCase. Update references to:

- `React.Fragment`
- `React.KeyedFragment`
- `React.Imported`
- `React.DynamicImported`
- `React.StrictMode`
- `React.Suspense`
- `React.Provider`
- `React.Consumer`

### Fable 5

From the [Fable.Cli changelog](https://github.com/fable-compiler/Fable/blob/main/src/Fable.Cli/CHANGELOG.md):

- **Compiler directive renamed from `FABLE_COMPILER_4` to `FABLE_COMPILER_5`.** Any `#if FABLE_COMPILER_4` blocks in your code will silently stop matching — search and replace before upgrading.
- **Requires the .NET 10 SDK.** Earlier SDKs are no longer supported.
- **Legacy Buildalyzer-based project cracker removed.** MSBuildCracker is the only parser; the `--legacyCracker` flag no longer exists.

(Fable 5 also reshapes the Python and Rust targets, but those don't apply to Elmish Land's JavaScript output.)

### vite 8

From the [Vite v7 → v8 migration guide](https://vite.dev/guide/migration):

- **Rollup replaced by Rolldown.** `build.rollupOptions` is now `build.rolldownOptions` (likewise `worker.rollupOptions` and `optimizeDeps.esbuildOptions` → `optimizeDeps.rolldownOptions`). A compatibility shim auto-converts the simple cases, but custom Rollup plugins may misbehave.
- **Default browser targets raised** — Chrome 107→111, Edge 107→111, Firefox 104→114, Safari 16.0→16.4. Override `build.target` if you still need to support older browsers.
- **CommonJS default-import semantics changed.** When importing a CJS module, `default` now resolves to `module.exports` in more cases than before; the shape of imported CJS-wrapped libraries may flip.
- **JS minification swapped to Oxc Minifier; CSS minification defaults to Lightning CSS.** `build.minify: 'esbuild'` is deprecated.
- **Removals**: `package.json` `browser`/`module` field sniffing; the object form of `output.manualChunks`; `build.commonjsOptions` is now non-functional.

### Fable.Elmish.HMR 9

The [HMR changelog](https://github.com/elmish/hmr/blob/master/src/CHANGELOG.md) does not document any explicit API breakages — the major was issued because it requires Fable 5 and Elmish 5. The internal change worth knowing: the React root is now cached internally to avoid the "multiple roots" warning, which is a runtime behaviour shift but should not require code changes.

## Dynamic version resolution during `init`

Previously, every scaffolded `Directory.Packages.props` and `package.json` had hardcoded versions baked into the CLI. That meant a new patch release of Feliz or vite required a new Elmish Land release before `init` would pick it up.

In 2.0, `init` resolves versions dynamically. The CLI keeps a list of *pinned majors* (Feliz 3, vite 8, Fable 5, …) and asks the NuGet and npm registries for the latest version matching each major at the moment you run `init`. New projects always start on the freshest available patch and minor releases.

## Upgrading an existing project: the new `upgrade` command

The migration story for existing projects is a single command:

```bash
dotnet tool update elmish-land --prerelease
dotnet elmish-land upgrade
```

`upgrade` brings your project to the dependency versions used by the current Elmish Land release. It will:

1. Detect the latest installed .NET SDK that meets the minimum required version and write it to `global.json`.
2. Update `Directory.Packages.props` so each managed `PackageVersion` lands on the latest version of its pinned major (Fable 5.x, Feliz 3.x, etc.).
3. Update `package.json` so `react`, `vite`, and friends move to the latest version of their pinned major.
4. Update Fable and any other managed dotnet tools in `.config/dotnet-tools.json` (or `dotnet-tools.json` in the project root, which is the new .NET SDK 10 default).
5. Regenerate the files under `.elmish-land/` so the new subscription-based URL handling and the vendored `Router.fs` are in place.
6. Run `dotnet restore` and `npm install` so the project is ready to build.

### What it preserves

The upgrade is conservative on purpose: it only touches what Elmish Land manages.

- Custom `PackageVersion` entries you've added to `Directory.Packages.props` are left alone.
- Packages in `package.json` that aren't part of the Elmish Land scaffold are kept as-is.
- Tools in `dotnet-tools.json` that you installed yourself are not modified.

### Back up your project before upgrading

`upgrade` modifies files in place, and the upstream libraries contain breaking changes (see [Breaking changes you may hit](#breaking-changes-you-may-hit)). Before running it, make sure your project is backed up — commit any pending work to git, or take a copy of the project directory. If your application doesn't compile after the upgrade, you can restore the previous state from your backup, address the breakages noted above, and try again.

## Quality of life fixes

- The interactive route-mode prompt during `init` no longer fights with the spinner — the prompt is shown first, then the spinner takes over for the actual scaffolding.
- The generated routing code no longer triggers F# warnings about unused query-parameter bindings on routes that don't declare any query parameters.
- The generated `sharedCommandToCmd` function no longer triggers a warning about an unused `msg` binding.

## Getting 2.0

If you already have Elmish Land installed:

```bash
dotnet tool update elmish-land --prerelease
dotnet elmish-land upgrade
```

Or for a fresh start:

```bash
dotnet new tool-manifest
dotnet tool install elmish-land --prerelease
dotnet elmish-land init
```

Plan a little time after upgrading to work through the breaking changes listed above — particularly the Feliz 3 ones, which are the most likely to require code edits.
